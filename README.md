# Layer10 Take-Home
## Grounded Long-Term Memory via Structured Extraction, Deduplication, and a Context Graph

---

## Corpus

**Enron Email Dataset** — the canonical benchmark for organizational memory at scale.

| Property | Value |
|----------|-------|
| Source | Kaggle: [wcukierski/enron-email-dataset](https://www.kaggle.com/datasets/wcukierski/enron-email-dataset?resource=download) |
| Format | `emails.csv` — columns: `file` (path), `message` (full RFC 2822 email) |
| Size | ~1.5 GB, ~517,000 emails |
| Origin | CMU / FERC public release of Enron internal email 1998–2002 |

**Why Enron:** It contains every pathology Layer10 will face in production — LDAP-formatted Exchange addresses, forwarding chains that create thousands of duplicate bodies, decisions followed by explicit reversals, multi-alias identities, and 4+ years of organizational history with real state changes. No synthetic dataset comes close.

**Reproduce the download:**
```bash
# Option 1 — Kaggle CLI
kaggle datasets download -d wcukierski/enron-email-dataset
unzip enron-email-dataset.zip          # produces emails.csv

# Option 2 — Direct browser download
# https://www.kaggle.com/datasets/wcukierski/enron-email-dataset?resource=download
```

## Quickstart

**Requirements:** Python 3.10+, zero external packages — pure stdlib only.

```bash
# 1. Unzip the submission
cd layer10/

# 2. Run the full pipeline
python3 pipeline.py --csv /path/to/emails.csv

# Optional: process a smaller slice first (fast, ~30 seconds)
python3 pipeline.py --csv emails.csv --limit 2000

# 3. Open the visualization — no server needed, just open the file
open viz/index.html          # macOS
xdg-open viz/index.html      # Linux
# Windows: double-click viz/index.html
```

The pipeline bakes all graph data directly into `viz/index.html` at the end of the run. No server, no separate data loading step.

## Visualization

Open `viz/index.html` directly in any browser after running the pipeline.

| Feature | Description |
|---------|-------------|
| **Graph canvas** | Force-directed layout of all entities. Blue = people, green = projects. Edge colors encode claim type. Pan, zoom, drag nodes. |
| **Entity panel** | Filterable list by type (All / People / Projects) or search by name/alias. Click to select. |
| **Claims tab** | All claims for selected entity, DISPUTED-first. Type badge, validity badge, predicate, excerpt, timestamps, confidence, evidence count. Click any claim to drill into evidence. |
| **Evidence tab** | All evidence pointers across the entity's claims. Exact excerpt, source ID, char offsets `[start:end]`, timestamp, confidence, extractor version. |
| **Merges tab** | Complete merge audit log — every artifact dedup, entity merge, and claim dedup event with reason and timestamp. |
| **Query bar** | Natural-language question → scored claims by entity match + TF-IDF + recency + confidence → top answer with validity annotation. |

## Key Design Decisions

**Zero external dependencies.** Pure Python stdlib: `csv`, `email`, `hashlib`, `re`, `json`, `collections`, `math`, `datetime`. Fully reproducible without pip, version conflicts, or network access.

**Deterministic extraction.** Regex pattern banks, not a model call. Every run on the same input produces identical output — essential for regression testing when the ontology changes.

**Evidence-first design.** No claim exists without an `EvidencePointer` carrying `source_id`, `excerpt`, `char_start`, `char_end`, `timestamp`, `confidence`, `extractor_ver`, and `schema_ver`. The quality gate enforces this at ingestion time.

**Bitemporal validity.** Every claim carries `valid_from` (event time) and `valid_until` (when superseded). Answers "what was true as of date D" as a simple range query.

**Reversible merges.** Every merge writes a structured entry to the merge log with merged IDs, canonical ID, reason, and timestamp. Any merge can be audited or undone.

## Honest Limitations

These are documented tradeoffs, not gaps. Each is covered in `WRITEUP.md`.

- **Identity resolution is incomplete.** LDAP-formatted Exchange addresses (`=ENRON/OU=NA/CN=...`) appear as separate entities. Full resolution needs a corporate directory — left as a documented production concern rather than papered over with heuristics.
- **Rule-based extraction misses implicit claims.** High precision, lower recall. Production path: LLM structured output with this layer as validation/fallback.
- **TF-IDF retrieval misses synonymy.** Production path: pgvector embeddings + BM25/cosine hybrid with RRF fusion.
- **In-memory graph.** Works to ~1M nodes. Production path: Postgres + pgvector + Apache AGE or Neo4j.
